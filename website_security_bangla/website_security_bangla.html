SSH Hardening — Remote Access নিরাপদ করোSecure Remote Access

# /etc/ssh/sshd_config পরিবর্তন করো: Port 2222 # Default 22 বদলাও (bot কম আসবে) PermitRootLogin no # Root login বন্ধ! PasswordAuthentication no # Password login বন্ধ → Key only PubkeyAuthentication yes # SSH Key ব্যবহার করো MaxAuthTries 3 # ৩ বারের বেশি try করতে পারবে না LoginGraceTime 20 # ২০ সেকেন্ডের মধ্যে login করতে হবে AllowUsers youruser # শুধু নির্দিষ্ট user Protocol 2 # SSH v2 only X11Forwarding no AllowTcpForwarding no # Restart করো: sudo systemctl restart sshd # SSH Key তৈরি (তোমার local machine-এ): ssh-keygen -t ed25519 -C "yourname@server" # Server-এ public key upload: ssh-copy-id -i ~/.ssh/id_ed25519.pub user@server

Firewall Setup — UFW (Ubuntu/DebianUbuntu/Debian)

# UFW install ও enable: sudo apt install ufw sudo ufw default deny incoming # সব incoming block sudo ufw default allow outgoing # সব outgoing allow # প্রয়োজনীয় port খুলো: sudo ufw allow 2222/tcp # SSH (তোমার custom port) sudo ufw allow 80/tcp # HTTP sudo ufw allow 443/tcp # HTTPS # Enable করো: sudo ufw enable sudo ufw status verbose # নির্দিষ্ট IP থেকে SSH allow (সবচেয়ে নিরাপদ): sudo ufw allow from YOUR.IP.ADDRESS to any port 2222 # Rate limiting (Brute force রোধ): sudo ufw limit ssh # ৩০ সেকেন্ডে ৬ বারের বেশি try করলে block

Fail2Ban — Brute Force Auto-BlockBrute Force Auto-Block

# Install: sudo apt install fail2ban # /etc/fail2ban/jail.local তৈরি করো: [DEFAULT] bantime = 3600 # ১ ঘণ্টা ban findtime = 600 # ১০ মিনিটে maxretry = 3 # ৩ বারের বেশি fail → ban banaction = ufw [sshd] enabled = true port = 2222 # তোমার SSH port [wordpress] enabled = true port = http,https filter = wordpress logpath = /var/log/auth.log maxretry = 5 # Start করো: sudo systemctl enable fail2ban sudo systemctl start fail2ban sudo fail2ban-client status # Status দেখো sudo fail2ban-client status sshd # Banned IP দেখো

🚨 Emergency Response — এই ক্রমে করো!Emergency Response — Do This in Order!

1
🔌 সাথে সাথে Isolate করোIsolate Immediately
Server-এর network connection বন্ধ করো অথবা Maintenance Mode চালু করো। Website offline করো যাতে visitor-রা ক্ষতিগ্রস্ত না হয়।Disconnect the server's network connection or enable Maintenance Mode. Take the website offline so visitors aren't harmed.
# WordPress Maintenance Mode: echo '<?php $upgrading = time(); ?>' > .maintenance # অথবা Cloudflare → Under Attack Mode চালু করো
2
🔑 সব Password পরিবর্তন করোChange All Passwords
cPanel, WordPress Admin, Database, FTP, Email, Hosting account — সব কিছুর password বদলাও। অন্য device থেকে করো।Change passwords for everything — cPanel, WordPress Admin, Database, FTP, Email, Hosting account. Do it from a different device.
3
📸 Evidence সংরক্ষণ করোPreserve Evidence
Log file, screenshot সেভ করো। কী হয়েছে তা বোঝার জন্য ও Legal প্রয়োজনে কাজে লাগবে।Save log files and screenshots. You'll need these to understand what happened and for any legal purposes.
sudo cp /var/log/auth.log /backup/auth_$(date +%Y%m%d).log sudo cp /var/log/apache2/access.log /backup/access_$(date +%Y%m%d).log
4
🔍 Malware Scan করোMalware Scan

5
🧹 Malware Remove করোRemove Malware
Infected file delete বা clean করো। WordPress-এর ক্ষেত্রে core files reinstall করা সবচেয়ে নিরাপদ।Delete or clean infected files. For WordPress, reinstalling core files is safest.

6
🔄 Clean Backup থেকে Restore করোRestore from a Clean Backup
Hack-এর আগের clean backup থেকে restore করা সবচেয়ে নিরাপদ। কিন্তু নিশ্চিত হও সেই backup-ও clean।Restoring from a clean backup before the hack is safest. But make sure that backup is also clean.
7
🔒 দুর্বলতা Fix করোFix the Vulnerability
কীভাবে hack হলো সেটা বুঝে সেই দুর্বলতা ঠিক করো। না হলে আবার hack হবে!Understand how the hack happened and fix that vulnerability. Otherwise, you'll get hacked again!
8
✅ Google Blacklist থেকে বের হওRemove from Google Blacklist
Google-এ flag হলে: Search Console → Security Issues → Review requested। Malware clean করার পরে request করো।If flagged by Google: Search Console → Security Issues → Request review. Do this after cleaning malware.

💾 Backup Strategy — সেরা বিমাBackup Strategy — Best Insurance
Backup না থাকলে সব হারানোর ঝুঁকি — 3-2-1 Rule follow করোWithout backup you risk losing everything — follow the 3-2-1 Rule

3-2-1 Backup Rule

3-2-1 Backup Rule:

3 copies of data    → Original + 2 backups
2 different media   → Server + External drive বা Cloud
1 offsite location  → Google Drive / S3 / Different datacenter

Example:
Production Server  →  Local Backup (same server)  →  Google Drive
     (live)              (daily)                     (weekly, offsite)
                                                         ↑
                                              এটা সবচেয়ে গুরুত্বপূর্ণ!

Backup Test — সবচেয়ে গুরুত্বপূর্ণ ধাপBackup Testing — The Most Important Step

⚠️ Untested Backup = No Backup!Untested Backup = No Backup!
প্রতি মাসে একবার backup restore করে দেখো এটা কাজ করছে কিনা। অনেকে মনে করে backup আছে, কিন্তু জরুরি সময়ে দেখা যায় backup corrupt বা incomplete। Once a month, try restoring your backup to verify it works. Many people think they have a backup, but in an emergency discover it's corrupt or incomplete.
